1. Introduction

Objectif du document : Fournir les bases de SQL utilisées dans le cours : lire et définir un schéma de tables, puis écrire les quatre opérations fondamentales (SELECT, INSERT, UPDATE, DELETE) depuis Python.

Le document sur les concepts et la modélisation présentait le modèle relationnel et son algèbre. SQL (Structured Query Language) en est la concrétisation : c’est le langage standard avec lequel on définit les tables d’une base de données et on manipule leur contenu. Les exemples de ce guide utilisent SQLite, un SGBD complet contenu dans une simple bibliothèque (incluse avec Python), idéal pour le développement et les applications embarquées. On le retrouve d’ailleurs dans une majorité de téléphones, de navigateurs et même de robots.

Les quatre opérations couvertes ici forment l’acronyme CRUD pour Create, Read, Update, Delete : créer, lire, modifier et supprimer des données. C’est le vocabulaire minimal de toute application qui persiste de l’information, et c’est souvent ainsi qu’on organise le code d’accès aux données : un module par table, regroupant ses quatre opérations. À chaque opération CRUD correspond une commande SQL (respectivement INSERT, SELECT, UPDATE et DELETE), et ce guide les présente dans cet ordre d’utilisation, après deux préalables : comment exécuter du SQL depuis Python, et comment lire le schéma d’une base existante.

Ce guide n’est pas une référence SQL complète : il couvre les outils de base, sans les fonctions d’agrégation ni les requêtes imbriquées. Les exemples reprennent le parc de robots mobiles du document sur les concepts : des robots affectés à des missions d’inspection, supervisées par des opérateurs.

2. Exécuter du SQL depuis Python

Avant d’écrire du SQL, il faut savoir comment l’acheminer vers la base. Avec le module sqlite3 de Python, on procède en trois temps : ouvrir une connexion vers le fichier de base de données, obtenir un curseur, puis soumettre des requêtes avec cursor.execute() :

import sqlite3

conn = sqlite3.connect("data/parc_robots.db")
conn.row_factory = sqlite3.Row   # accès aux colonnes par nom

cursor = conn.cursor()
cursor.execute("SELECT * FROM robots WHERE serie = ?", (serie,))
# ... exploiter le résultat ...

conn.commit()
conn.close()

Le conn.commit() n’est pas décoratif : c’est lui qui rend les modifications permanentes. Une écriture sans commit est invisible pour les autres connexions et perdue à la fermeture. Si vos insertions « disparaissent », c’est le premier suspect. Son pendant, conn.rollback(), annule les modifications depuis le dernier commit, ce qui permet d’abandonner proprement une séquence d’opérations interrompue par une erreur. En pratique, on encapsule souvent cette plomberie (connexion, commit ou rollback selon le succès, fermeture) dans un gestionnaire de contexte, pour que chaque fonction se réduise à un bloc with. Vous rencontrerez ce patron dans le projet.

Le deuxième point fondamental est le passage de paramètres. Une requête contient presque toujours des valeurs qui proviennent du programme (un numéro de série, un résultat…​). On ne les insère jamais dans la chaîne SQL soi-même :

# DANGEREUX - ne faites jamais ceci :
cursor.execute(f"SELECT * FROM robots WHERE serie = '{serie}'")

# Correct : un ? par valeur, et un tuple de valeurs en second argument
cursor.execute("SELECT * FROM robots WHERE serie = ?", (serie,))

La version avec ? laisse sqlite3 encoder les valeurs correctement (guillemets, types, caractères spéciaux). La version avec interpolation de chaîne, en plus d’être fragile, ouvre la porte aux injections SQL : si la valeur vient d’un utilisateur, rien ne l’empêche d’y glisser du SQL malveillant qui sera exécuté tel quel. C’est l’une des vulnérabilités les plus exploitées au monde. Remarquez la virgule dans (serie,) : le second argument doit être un tuple, même pour un seul paramètre.

Enfin, quelques attributs du curseur servent à exploiter le résultat d’une requête : cursor.fetchone() retourne la prochaine rangée du résultat (ou None s’il n’y en a pas), cursor.fetchall() retourne la liste de toutes les rangées, cursor.lastrowid donne l’identifiant auto-généré de la dernière insertion, et cursor.rowcount indique combien de rangées ont été touchées par un UPDATE ou un DELETE. Avec row_factory = sqlite3.Row (configurée plus haut), les colonnes d’une rangée sont accessibles par leur nom (row["modele"]) plutôt que par leur position. Profitez-en, le code y gagne en lisibilité.

3. Définir les tables : CREATE TABLE

Le fichier de schéma est le contrat que toutes les requêtes doivent respecter : il faut savoir le lire pour travailler avec une base existante, et savoir l'écrire dès qu’on conçoit ou normalise un modèle de données. Décomposer une table qui viole la 3FN, c’est précisément remplacer un CREATE TABLE par plusieurs. Voici comment le schéma du parc de robots du document sur les concepts s’écrit en SQL :

CREATE TABLE IF NOT EXISTS robots (
    serie           TEXT PRIMARY KEY,
    modele          TEXT NOT NULL,
    mise_en_service TEXT NOT NULL,
    charge_utile    REAL NOT NULL,

    CHECK (charge_utile > 0)
);

CREATE TABLE IF NOT EXISTS affectations (
    serie            TEXT NOT NULL,
    code             TEXT NOT NULL,
    date_affectation TEXT NOT NULL,
    resultat         TEXT,

    PRIMARY KEY (serie, code),
    FOREIGN KEY (serie) REFERENCES robots(serie),
    FOREIGN KEY (code) REFERENCES missions(code),
    CHECK (resultat IN ('succes', 'echec', 'abandon'))
);

Chaque ligne déclare une colonne avec son type (TEXT, INTEGER, REAL…​) et ses contraintes. C’est la traduction directe des notions du document sur les concepts. PRIMARY KEY désigne la clé primaire, simple (serie) ou composée ((serie, code), héritée de l’identifiant du lien Affectation). NOT NULL rend la colonne obligatoire, et resultat ne l’est pas : une affectation en cours n’a pas encore de résultat. UNIQUE interdit les doublons sur une colonne qui n’est pas la clé, et DEFAULT fournit une valeur en cas d’omission. FOREIGN KEY déclare une clé étrangère vers une autre table, et CHECK restreint le domaine des valeurs admises. Ainsi, le SGBD lui-même rejettera un résultat inconnu ou une charge utile négative, dernière ligne de défense derrière la validation que le code applicatif fait en amont.

Deux compléments qu’on rencontre couramment dans un fichier de schéma. Les identifiants artificiels : lorsqu’une table n’a pas de clé naturelle, on déclare une colonne INTEGER PRIMARY KEY AUTOINCREMENT et le SGBD génère lui-même la valeur à chaque insertion ; dans ce cas, on ne fournit pas cette colonne dans les INSERT, et on récupère la valeur générée via cursor.lastrowid. Les index : une déclaration comme CREATE INDEX idx_affectations_serie ON affectations(serie) ne change rien au contenu de la base, mais accélère les recherches sur la colonne visée. Un détail d’implémentation qui n’affecte pas vos requêtes, mais qui peut faire une grande différence de performance sur de grandes bases de données.

Le schéma lui-même se manipule avec les compagnons de CREATE TABLE : DROP TABLE supprime une table et tout son contenu (utile pour remplacer une table mal conçue par sa version normalisée, après avoir migré les données !), et ALTER TABLE apporte des modifications ponctuelles comme l’ajout d’une colonne. Enfin, pour initialiser une base à partir d’un fichier de schéma complet, cursor.executescript() exécute d’un coup un texte contenant plusieurs commandes SQL séparées par des ;, contrairement à execute(), qui n’en accepte qu’une seule.

4. SELECT : lire (le R de CRUD)

SELECT est la traduction SQL des opérateurs de l’algèbre relationnelle : on y retrouve la projection (le choix des colonnes) et la sélection (la clause WHERE). Sa forme de base :

SELECT code, date_affectation, resultat
FROM affectations
WHERE serie = ? AND resultat = 'echec'
ORDER BY date_affectation DESC;

La liste qui suit SELECT énumère les colonnes voulues ; les demande toutes, ce qui est acceptable en exploration mais fragile dans du code (l’ordre et le nombre de colonnes peuvent changer avec le schéma. Nommer les colonnes rend la requête plus robuste). WHERE filtre les rangées avec des conditions combinables par AND, OR et NOT ; les opérateurs usuels sont =, !=, <, <=, >, >=, plus IN (…​) pour tester l’appartenance à une liste, LIKE pour les motifs de texte ('UGV-%'), et IS NULL / IS NOT NULL pour tester l’absence de valeur. Un simple = NULL ne fonctionne pas, NULL n’étant égal à rien, pas même à lui-même. ORDER BY trie le résultat (DESC pour décroissant) ; sans lui, l’ordre des rangées n’est *pas garanti. C’est une conséquence directe du fait qu’une relation est un ensemble.

Un patron récurrent dans les fonctions de listage : construire la clause WHERE selon les filtres réellement demandés. L’astuce du WHERE 1=1 (une condition toujours vraie) permet d’ajouter chaque filtre avec un simple AND, sans cas particulier pour le premier :

query = "SELECT serie, code, date_affectation, resultat FROM affectations WHERE 1=1"
params = []
if serie is not None:
    query += " AND serie = ?"
    params.append(serie)
if resultat is not None:
    query += " AND resultat = ?"
    params.append(resultat)
query += " ORDER BY date_affectation DESC"
cursor.execute(query, params)

Notez que seuls les fragments fixes de SQL sont assemblés par concaténation ; les valeurs, elles, passent toujours par ?.

Un mot enfin sur la jointure, qui sert dès qu’une réponse combine plusieurs tables, par exemple afficher chaque affectation avec le modèle du robot concerné. La normalisation a réparti l’information entre robots et affectations ; JOIN …​ ON la recompose à la demande, exactement comme l’opérateur ⋈ de l’algèbre relationnelle :

SELECT r.serie, r.modele, a.code, a.resultat
FROM robots r
JOIN affectations a ON a.serie = r.serie
ORDER BY r.serie;

Les alias (r, a) abrègent les noms de tables, et la condition ON apparie chaque affectation au robot dont la clé correspond à sa clé étrangère.

5. INSERT : créer (le C de CRUD)

INSERT ajoute une rangée dans une table. On nomme explicitement les colonnes fournies ; les colonnes omises reçoivent leur valeur DEFAULT ou NULL :

cursor.execute(
    """
    INSERT INTO affectations (serie, code, date_affectation)
    VALUES (?, ?, ?)
    """,
    (serie, code, date_affectation),
)

Ici, resultat n’est pas fourni et restera NULL. L’affectation vient d’être créée, la mission n’est pas terminée. Si une contrainte du schéma est violée, par exemple avec un numéro de série inexistant (clé étrangère), un couple (serie, code) déjà présent (clé primaire), ou un résultat hors de la liste du CHECK, sqlite3 lève une exception IntegrityError : c’est la base qui défend sa propre cohérence. Et pour les tables à identifiant artificiel AUTOINCREMENT, c’est immédiatement après l’INSERT qu’on récupère cursor.lastrowid pour le retourner à l’appelant.

Pour insérer plusieurs rangées d’un coup (comme dans un fichier de données d’exemple), VALUES accepte plusieurs tuples séparés par des virgules ; côté Python, cursor.executemany() rend le même service en répétant une requête sur une liste de tuples de paramètres.

6. UPDATE : modifier (le U de CRUD)

UPDATE modifie les colonnes listées dans SET pour toutes les rangées qui satisfont le WHERE. Par exemple, pour consigner le résultat d’une mission terminée on pourrait utiliser cette requête :

cursor.execute(
    "UPDATE affectations SET resultat = ? WHERE serie = ? AND code = ?",
    (resultat, serie, code),
)
modifie = cursor.rowcount > 0

Deux réflexes à acquérir. D’abord, toujours vérifier le WHERE deux fois : un UPDATE sans clause WHERE est syntaxiquement valide et modifie toutes les rangées de la table. C’est l’erreur classique qui marque toutes les affectations du parc comme « échec ». Ensuite, exploiter cursor.rowcount : s’il vaut 0, aucune rangée ne correspondait au filtre (l’affectation n’existe pas), ce que la fonction appelante voudra généralement signaler. Prenez aussi l’habitude de valider les valeurs avant d’exécuter la requête (le résultat est-il bien dans ['succes', 'echec', 'abandon'] ?) et de lever ValueError sinon : le CHECK du schéma rejetterait la valeur de toute façon, mais une validation explicite en Python produit un message d’erreur clair plutôt qu’une exception de bas niveau.

Lorsque les colonnes à modifier varient selon les paramètres reçus, on assemble la clause SET dynamiquement avec la même discipline que pour le WHERE optionnel : fragments fixes concaténés, valeurs passées par ?.

7. DELETE : supprimer (le D de CRUD)

DELETE retire de la table toutes les rangées qui satisfont le WHERE :

cursor.execute(
    "DELETE FROM affectations WHERE serie = ? AND code = ?",
    (serie, code),
)
supprime = cursor.rowcount > 0

Le même avertissement s’applique avec encore plus de force : DELETE FROM affectations sans WHERE vide la table entière, sans confirmation ni retour en arrière une fois le commit passé. Comme pour UPDATE, rowcount indique si une rangée a réellement été supprimée.

La suppression interagit avec les clés étrangères : que deviennent les affectations d’un robot qu’on retire du parc ? Selon la configuration de la contrainte, le SGBD peut refuser la suppression tant que des affectations y réfèrent, ou les supprimer en cascade (ON DELETE CASCADE). Dans tous les cas, la règle de fond est la même : la base ne doit jamais contenir de clé étrangère qui pointe vers une rangée disparue.

8. Récapitulatif

Tableau 1. Correspondance entre les opérations CRUD, SQL et sqlite3
Opération SQL Côté Python

Create

INSERT INTO t (…​) VALUES (?, …​)

cursor.lastrowid pour l’ID généré

Read

SELECT …​ FROM t WHERE …​ ORDER BY …​

fetchone() / fetchall()

Update

UPDATE t SET col = ? WHERE …​

cursor.rowcount pour le nombre touché

Delete

cursor.rowcount pour le nombre touché

La démarche est toujours la même : lire le schéma pour connaître les colonnes et leurs contraintes, ouvrir la connexion (ou réutiliser le gestionnaire prévu à cet effet), écrire le SQL avec des ? pour toutes les valeurs, exploiter le résultat (fetchone/fetchall, lastrowid, rowcount), et s’assurer que le commit a lieu.

Enfin, pour expérimenter une requête avant de l’intégrer au code, l’interpréteur en ligne de commande de SQLite permet de l’essayer directement contre la base. Dans l’environnement du cours, géré par uv, on l’invoque ainsi :

uv run sqlite3 data/gro321.db

On peut alors taper des requêtes SQL interactivement (terminées par ;), et quelques commandes utiles propres à l’outil : .tables liste les tables, .schema robots affiche le CREATE TABLE d’une table, et .quit quitte l’interpréteur.
